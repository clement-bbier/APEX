## Notes on skipped / failing benches

- **Risk chain E2E** — skipped in this PR. The orchestrator requires an
  injected mock stack (`FailClosedGuard`, `CBEventGuard`,
  `CircuitBreaker`, `MetaLabelGate`, `RiskDecisionBuilder`, plus an
  async context loader). Assembling that mock stack touches
  `services/` internals which this PR deliberately excludes. The
  benchmark file (`bench_risk_chain.py`) contains the activation steps
  for Phase B. Prior integration-test data (`tests/integration/s05/…`)
  shows the full 6-step chain runs in 2–4 ms end-to-end in-process,
  dominated by `await` roundtrips to Redis — clearly not a Rust-port
  candidate.
- **Kyle lambda with `kyle_window=250`** — originally errored at
  `features/calculators/cvd_kyle.py:138` because the default
  `kyle_zscore_lookback=252` left no z-score history buffer for
  `kyle_window=250` (calculator enforces `lookback >= 2 * kyle_window`).
  **Fixed in #239** (2026-04-22): the benchmark now sets
  `kyle_zscore_lookback = max(252, kyle_window * 2)`, and the calculator
  error message includes the offending values plus the recommendation.
  Both `w=250` rows in the table above were captured under that fix.

## Rust port prioritization — Top 3 candidates for Phase B

Ranked by **expected p99 reduction on the tick-path signal budget**:

1. **Kyle lambda rolling OLS** (`features/calculators/cvd_kyle.py`) —
   ranking #1 because (a) it's the per-tick loop with the highest
   absolute cost per call, (b) the underlying math (rolling OLS +
   rolling z-score) is trivial in `ndarray` / `rayon`, and (c) z-score
   history can live in a `VecDeque` with O(1) push/pop instead of
   per-tick polars window rebuilds. **Expected speedup: 10–20×**.
   Target Rust p99 for 20 000 ticks: 400 ms (Python: 9 s).
2. **OFI multi-window** (`features/calculators/ofi.py`) — #2 because
   OFI fires on every tick and the trade-based fallback is exactly
   the code path that will run in production for most symbols.
   Three-window loop + tanh normalization maps cleanly to SIMD.
   **Expected speedup: 5–15×**. Target Rust p99 for 100 000 ticks:
   1.2 s (Python: 12 s).
3. **CVD divergence** (`features/calculators/cvd_kyle.py`, CVD half) —
   #3 only because it's the same calculator as Kyle: porting Kyle
   also ports CVD's rolling divergence almost for free. Cumulative
   sum + rolling OLS on (price, cvd) is a natural fit for the same
   Rust crate. **Expected speedup: 3–8×** on the CVD-only portion,
   but compounds with Kyle's gain when the calculator runs as one.

Everything else — HAR-RV (batch, low frequency), Kelly sizer
(sub-microsecond), risk chain (Redis-bound) — stays in Python.

### Expected aggregate p99 latency reduction

If the top-3 ports land in Phase B and cut each by their estimated
speedups, the aggregate tick-path signal latency (OFI + CVD + Kyle on
a 10 000-tick window) drops from roughly **4.9 s → ~0.5–0.7 s** on
this hardware, a **~7–10× improvement** on the full tick-path
microstructure pack. This is what unlocks strategy research loops that
iterate at full-day resolution (100 000 ticks) in under a minute
rather than 12+ seconds per iteration.

### Rationale (short form)

- The "high-priority" candidates are all (a) on the tick hot path,
  (b) dominated by a pure numeric inner loop, (c) small enough in
  scope that each is a ~300-line Rust file. That matches what
  `apex_mc` and `apex_risk` already do for Monte Carlo and exposure.
- The "don't port" list is not lazy: Kelly is already sub-microsecond
  once Decimal is warm, HAR-RV is a daily-refresh feature, and the
  risk chain is asynchronous I/O — none of these benefit from Rust.
